ERRORS AND BUGS REPORT: blocking_mpmc_unbounded
=====================================================

1. [CRITICAL] Data Race on 'size_q'
   - Location: include/blocking_mpmc_unbounded/defs.hpp (line 25) and throughout impl.hpp.
   - Issue: 'size_q' is a standard 'size_t' and is modified/read across different threads using different mutexes ('head_mutex' and 'tail_mutex'). This constitutes a data race. 
   - Consequence: Undefined behavior. The queue size may become corrupted, leading to incorrect 'empty()' reports or 'wait_and_pop' blocking indefinitely or returning prematurely.
   - Fix: Change 'size_q' to 'std::atomic<size_t>'.

2. [BUG] Thread-Safety of 'empty()' and 'size()'
   - Issue: These methods read 'size_q' without any synchronization or while only holding one of the two mutexes. Because 'push' only locks 'tail_mutex' and 'pop' only locks 'head_mutex', a call to 'empty()' from a popping thread does not synchronize with the 'size_q++' in a pushing thread.
   - Fix: Making 'size_q' atomic will resolve the visibility and race issues for these methods.

3. [FLAW] Notification Pessimization
   - Location: include/blocking_mpmc_unbounded/impl.hpp (line 27 and 137).
   - Issue: 'cond.notify_one()' is called while holding 'tail_mutex'. While correct, it can cause a "hurry up and wait" scenario where the woken thread tries to acquire a mutex that is still held by the notifier.
   - Suggestion: Move 'cond.notify_one()' outside the 'std::lock_guard' scope.

4. [FLAW] Inefficient Memory Allocation
   - Issue: Every 'push' or 'emplace_back' performs two dynamic allocations: one for 'std::make_unique<node>()' and one for 'std::make_shared<T>()'.
   - Suggestion: Consider storing 'T' directly inside the 'Node' (perhaps using 'std::optional<T>' or 'std::aligned_storage') to reduce allocations to one per push.

5. [POTENTIAL BUG] Copy-Assignment Dependency
   - Issue: 'wait_and_pop(T &value)' and 'try_pop(T &value)' use 'value = *(ret->data)'. This performs a copy assignment.
   - Consequence: This will fail to compile for move-only types (like 'std::unique_ptr') and is inefficient for large objects.
   - Fix: Use 'value = std::move(*(ret->data))' to support move-only types and improve efficiency.

6. [USABILITY] Namespace Pollution in queue.hpp
   - Issue: 'using namespace tsfqueue::__impl;' in a header file is generally discouraged as it pollutes the namespace of any file that includes it.

REQUIRED STATIC ASSERTS:
========================

Add these to your class definition in defs.hpp to ensure type compatibility:

static_assert(std::is_copy_constructible_v<T> || std::is_move_constructible_v<T>, 
              "T must be copyable or movable to be pushed into the queue.");

static_assert(std::is_copy_assignable_v<T> || std::is_move_assignable_v<T>, 
              "T must be copy-assignable or move-assignable to be popped into a reference.");

static_assert(!std::is_reference_v<T>, 
              "Queue cannot store reference types.");
